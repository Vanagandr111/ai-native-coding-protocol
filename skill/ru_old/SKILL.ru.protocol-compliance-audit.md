## 1. Триггеры

Запускай этот скилл, когда пользователь говорит:

- «сделай код-ревью»
- «проверь по протоколу»
- «проверь качество кода»
- «audit»
- «compliance check»
- «пройдись по чеклисту»
- «проверь по ai-native-coding-protocol»
- «сделай» (в контексте проверки кода)

Также запускай **автоматически после фиксации багов**, если touchнула больше 2 файлов.

## 2. Процедура проверки

### Шаг 0 — Load skills

```python
# Обязательно:
skill_view(name='ai-native-coding-protocol')
skill_view(name='protocol-compliance-audit')  # этот файл

# По необходимости:
skill_view(name='bleed-stop')  # если есть patch() или write_file() в плане
```

### Шаг 1 — Собери контекст

```bash
# 1. Какие файлы трогались?
git status --short

# 2. Если гита нет — ls -la и смотрим изменения вручную
ls -la изменённые_файлы

# 3. Сколько строк в каждом файле?
wc -l изменённые_файлы

# 4. Структура файла
grep -n "^def \|^class \|^# ── \|^# -- \|^# [A-Z]" file.py | head -40
```

### Шаг 2 — Пройдись по чеклисту (раздел 4)

Для каждого пункта:
1. Прочитай правило из `ai-native-coding-protocol` (в памяти)
2. Проверь файл на соответствие
3. Запиши PASS / FAIL / WARN
4. Для FAIL — опиши конкретное нарушение

### Шаг 3 — Сформируй отчёт (раздел 5)

### Шаг 4 — Исправь нарушения (раздел 6)

Правило: **одно нарушение = один patch**. Не делай 5 исправлений одним patch/write_file.

### Шаг 5 — Повторная проверка

После исправлений пройдись по FAIL-пунктам ещё раз, чтобы убедиться что они стали PASS.

## 3. Grep-команды для быстрой проверки

```bash
# Паспорта функций
grep -c "# intent:" file.py
grep -c "^def " file.py
# Если #intent: < def — есть непокрытые функции

# AI Navigation Layer
grep -c "# module-purpose:" file.py
grep -c "# entry-points:" file.py
grep -c "# main-flows:" file.py

# Temporary code
grep -n "TEMP\|FIXME\|TODO\|HACK\|XXX\|WORKAROUND" file.py

# Magic values (числа кроме 0, 1, -1)
grep -nP '(?<!= )\b(?:[2-9]|[1-9]\d{2,})\b' file.py | grep -v "offset\|limit\|timeout\|max_\|min_\|port\|count\|rows\|cols\|line_\|step\|version\|padding\|gap\|index\|row_idx\|ci\."

# Hardcoded paths
grep -n '"/mnt\|"C:\\\\|"/tmp\|os\.environ\["\|environ\.get("' file.py

# Импорты (зависимости)
grep "^import \|^from " file.py

# Все public функции
grep -n "^def [^_]" file.py

# Все private функции
grep -n "^def _" file.py

# sidefx annotation
grep -n "# sidefx:" file.py

# typing — параметры
grep -n "^def .*," file.py | grep -v "# "  # функции с >1 параметром
```

## 4. Пункты протокола — что проверять

### #0 — Skill loaded first
- **Проверка:** `skill_view(name='ai-native-coding-protocol')` вызван ДО изменений?
- **PASS:** был загружен до первого patch/write_file
- **FAIL:** начал править код без загрузки
- **Fix:** загрузить сейчас, проверить что уже изменённое соответствует протоколу, зафиксировать нарушение в отчёте

### #1 — Reuse first
- **Проверка:** Новая функция дублирует существующую? Есть похожие по имени/смыслу?
- **PASS:** расширил существующую или новая — действительно новая
- **FAIL:** создана функция, дублирующая ~70% существующей
- **Fix:** удалить новую, расширить существующую параметром/опциями

### #2 — GIT backup
- **Проверка:** `git status` — есть незакоммиченные изменения? Был сделан стэш/коммит до правок?
- **PASS:** git commit или stash сделан до правок, либо изменений нет
- **WARN:** git есть, правки есть, стэша/коммита не было
- **FAIL:** гита нет, правки есть, бекапа нет
- **Fix:** `git stash push -m "pre-audit $(date +%Y%m%d_%H%M%S)"` или `cp file file.bak`

### #3 — Anti-Duplication
- **Проверка:** Поиск дублирующихся блоков (>5 строк)
- **PASS:** нет дублирования
- **FAIL:** есть дублирующиеся логические блоки
- **Fix:** вынести в функцию, параметризовать

### #4 — Balanced Architecture
- **Проверка:** Размер файла разумен для задачи?
  - <50 строк: слишком мало для отдельного файла
  - 50-200: helpers/small module
  - 200-1200: normal module
  - >1200: рассмотреть разделение
- **PASS:** размер соответствует
- **WARN:** >1200 строк, рассмотреть рефакторинг
- **Fix:** вынести unrelated функции в отдельный файл

### #5 — Semantic Cluster
- **Проверка:** Все функции в файле относятся к одному домену?
- **PASS:** один файл = один домен
- **FAIL:** в файле перемешаны LAMMPS + UI + HTTP + DB
- **Fix:** разделить по доменам

### #6 — Architecture Limits
- **Проверка:** Нет файлов по 5-10 строк? Нет смешивания слоёв?
- **PASS:** всё нормально
- **FAIL:** есть микро-файлы или смешивание слоёв

### #7 — AI-Readable Code
- **Проверка:** Стабильные паттерны, консистентная структура?
- **PASS:** однородный стиль
- **FAIL:** в одном файле snake_case и camelCase, разные стили докстрингов
- **Fix:** привести к единому стилю

### #8 — File Structure
- **Проверка:** Соблюдён порядок? imports → constants → types → config → services → helpers → main → exports
- **PASS:** порядок соблюдён
- **FAIL:** константы после функций, импорты в середине
- **Fix:** переставить секции

### #9 — Semantic Function Passport
- **Проверка:** У каждой public функции есть passport (`# intent:`, `# allies:`, `# sidefx:`, `# purity:`, `# errors:`)?
- **PASS:** у всех public функций полный паспорт
- **WARN:** у private функций без паспорта (допустимо для >3 строчных утилит)
- **FAIL:** у public функции нет паспорта
- **Fix:** добавить паспорт

### #10 — Naming Protocol
- **Проверка:** `verb_noun` для функций, PascalCase для классов, UPPER_SNAKE для констант?
- **PASS:** соответствует
- **FAIL:** есть нарушения
- **Fix:** переименовать

### #11 — Explicit Separation
- **Проверка:** sidefx: annotation правильный? Не скрыты побочные эффекты?
- **PASS:** всё явно
- **FAIL:** функция помечена pure, но пишет в файл/БД
- **Fix:** поправить annotation или разделить

### #12 — Typing Rules
- **Проверка:** >3 параметров → DTO/dataclass?
- **PASS:** либо ≤3 параметра, либо есть DTO
- **WARN:** >3 параметров без DTO (допустимо для legacy/helpers)
- **Fix:** создать dataclass

### #13 — No Magic Values
- **Проверка:** Есть числа кроме 0/1/-1, строки-пути, таймауты?
- **PASS:** все значения вынесены в константы
- **FAIL:** магические числа/строки
- **Fix:** вынести в константу

### #14 — Dependency Discipline
- **Проверка:** Каждая внешняя зависимость оправдана?
- **FAIL:** импорт библиотеки ради одной функции, которую легко написать самому
- **WARN:** опциональная зависимость без graceful fallback
- **Fix:** добавить try/except или удалить

### #15 — Fail Fast
- **Проверка:** Валидация на входе? Понятные ошибки? Нет silent failures?
- **PASS:** всё валидируется, ошибки информативны
- **FAIL:** silent return без сообщения, непонятные исключения
- **Fix:** добавить raise ValueError(...) или print(...) + return

### #16 — Pure Function Preference
- **Проверка:** pure функции annotated correctly?
- **FAIL:** функция помечена pure, но читает/пишет внешнее состояние
- **Fix:** поправить annotation или имплементацию

### #17 — Single Abstraction Level
- **Проверка:** В одной функции нет SQL + HTTP + бизнес-логики?
- **PASS:** каждый уровень разделён
- **FAIL:** смешивание
- **Fix:** разделить на helper-функции

### #18 — Anti-Hardcode
- **Проверка:** Нет хардкода путей, URL, токенов, таймаутов?
- **PASS:** всё через параметры/config/PROJDIR
- **FAIL:** захардкожено
- **Fix:** вынести в параметр/константу

### #19 — Temporary Code
- **Проверка:** Есть `# TEMP:`, `# FIXME:`, `# TODO:` блоки?
- **PASS:** нет
- **WARN:** есть TEMP с `# reason:` и `# remove-after:`
- **FAIL:** есть TEMP без даты удаления
- **Fix:** добавить дату или убрать

### #20 — AI Navigation Layer
- **Проверка:** Есть `# module-purpose:`, `# entry-points:`, `# main-flows:`, `# related-modules:`?
- **PASS:** все 4 поля есть
- **WARN:** неполный (2-3 поля)
- **FAIL:** нет ни одного
- **Fix:** добавить

### #21 — Architectural Manifest
- **Проверка:** Для large feature есть предварительная структура?
- **PASS:** manifest есть для крупных изменений
- **FAIL:** large feature без manifest
- **N/A:** мелкие правки

### #22 — Predictability Rule
- **Проверка:** Одинаковые задачи решены одинаковым паттерном?
- **PASS:** консистентно
- **FAIL:** в одном месте `if/else`, в другом `match/case` для той же логики

### #23 — AI Memory Anchors
- **Проверка:** Есть `# critical:`, `# architecture:`, `# related:`?
- **PASS:** есть
- **FAIL:** нет (для ключевых файлов)
- **N/A:** для мелких helpers

## 5. Шаблон отчёта

После проверки возвращай отчёт в таком формате:

```
## 📋 Аудит по AI-Native Coding Protocol

**Файл(ы):** path/to/file.py
**Git статус:** clean | dirty | no-git

### 🔴 FAIL (требуют исправления)
| # | Пункт | Проблема | Исправлено? |
|---|-------|----------|-------------|
| 2 | GIT backup | Не сделан стэш до правок | ✅ / ⬜ |
| ... | ... | ... | ... |

### 🟡 WARN (рекомендации)
| # | Пункт | Замечание |
|---|-------|-----------|
| ... | ... | ... |

### ✅ PASS (всего X из Y)
| # | Пункт | Статус |
|---|-------|--------|
| 1 | Reuse first | ✅ |
| ... | ... | ✅ |

### ⛑️ Bleed-stop audit (если были patch/write_file)
| Операция | Риск | Статус |
|----------|------|--------|
| patch() на ... | ... | ✅ / ⚠️ / 🔴 |

### Итого
- PASS: X
- WARN: Y
- FAIL: Z (из них исправлено: W)
- **Вердикт:** ✅ чисто / ⚠️ мелкие недочёты / 🔴 нужны правки
```

## 6. Типовые исправления

### Если нет паспорта (#9)

```python
# Добавить перед функцией:
# intent: <что делает>
# allies: <кто вызывает>
# sidefx: none|io|state|db
# purity: pure|impure
# errors: <как падает>
```

### Если нет AI Navigation Layer (#20)

В начало файла (после docstring или в начало):
```python
# module-purpose: <назначение модуля>
# entry-points: <точки входа>
# main-flows: <основные потоки>
# related-modules: <связанные модули>
```

### Если magic value (#13)

Замени захардкоженное число на константу:
```python
# В секцию констант:
COLUMN_OFFSET = 65
# В коде:
chr(COLUMN_OFFSET + ci)
```

### Если silent failure (#15)

Добавить сообщение:
```python
except ImportError:
    print('  MODULE: skipped (lib not available)')
    return
```

### Если нет git backup (#2)

```bash
git stash push -m "pre-audit $(date +%Y%m%d_%H%M%S)"
# или
cp file.py file.py.bak
```

## 7. Что НЕ проверять

- **Тестовые файлы** — на них протокол не распространяется (`Does NOT apply to: pure test/TDD cycles`)
- **Конфиги** — `.yaml`, `.json`, `.toml`
- **Markdown/README** — протокол только для кода
- **Сгенерированный код** — proto, swagger, graphql
- **Файлы < 30 строк** — если это лёгкий helper/shell
- **Чистые тесты** — test_*.py без бизнес-логики

## 8. Пример полного аудита

Смотри предыдущую сессию с `lmp_helper.py` — live-пример прохождения по всем 23 пунктам с grep-командами, исправлениями и финальным отчётом.

Ключевые уроки из того аудита:
1. Загружай скилл ДО первого patch, не после — иначе #0 FAIL
2. Проверяй git STATUS перед правками — иначе #2 FAIL
3. Grep ищи magic values руками — `chr(65 + ci)` не очевиден
4. Паспорт `# errors:` должен описывать РЕАЛЬНОЕ поведение, не желаемое
5. После каждого исправления — `git diff` чтобы проверить что не сломал
