# 0. PRIORITY ORDER

1. reuse existing code
2. extend existing code
3. create new code

# 1. Главный закон разработки
Цель ИИ — не генерировать новый код любой ценой.
Создание нового кода — крайняя мера.
Перед созданием:
- проверить existing + allies/tags/intent
- переиспользовать или расширить
Дублирование запрещено

# 2. GIT BACKUP PROTOCOL
Если доступен git и пользователь разрешил commits, перед изменениями создать git checkpoint.
Если commits не разрешены, перед изменениями проверить и сообщить dirty state.

Правила:
- один commit = одна логическая задача
- не смешивать разные изменения в одном commit
- commit message должен кратко описывать выполненные изменения
- всегда сохранять рабочее состояние перед изменениями

# 3. Anti-Duplication Protocol
Перед созданием новой функции ИИ обязан выполнить:
## Проверка:
- похожих названий
- allies
- tags
- intent
- semantic similarity
- существующих контрактов
Если функция уже покрывает ≥70% задачи:
- расширять существующую
- а не создавать новую

# 4. Balanced Architecture Rule
ИИ запрещено:
- создавать спагетти-код
- создавать тысячи микрофайлов
- дробить архитектуру без причины

## Основной принцип:
Размер модуля должен соответствовать сложности задачи.

# 5. Правило смыслового кластера
Один файл = одна предметная область.
Разрешено хранить вместе:
- связанные функции
- связанные типы
- связанные классы
- связанные сервисы

Если они относятся к одной ответственности.

# 6. Ограничения архитектуры
## Нельзя:
- смешивать UI, бизнес-логику и инфраструктуру
- создавать файл ради 5-10 строк
## Рекомендуется:
- 3–10 публичных методов на модуль
- один уровень абстракции внутри функции
- предсказуемая структура проекта

# 7. AI-Readable Code
Код должен быть оптимизирован:
- не только для человека
- но и для чтения ИИ
ИИ лучше понимает:
- одинаковые паттерны
- стабильную структуру
- предсказуемость
- повторяемые правила

# 8. Обязательная структура файла
Порядок секций фиксирован:
1. imports
2. constants
3. types/interfaces/models
4. config
5. services
6. helpers
7. main logic
8. exports
Менять порядок запрещено без причины.

# 9. Semantic Function Passport
Каждая публичная функция/класс обязана иметь этот докблок для AI-навигации:
` ` `python
# intent: [человеческое описание задачи]
# allies: [синонимы_для_поиска, через, запятую]
# sidefx: [none | io | db | http | state]
# purity: [pure | impure]
# errors: [список выбрасываемых исключений]
` ` `
*Дополнительные теги (по необходимости):* `# mutates:`, `# related:`, `# temp-reason: [почему_временное] remove-after: [условие]`.


# 10. Naming Protocol
Названия обязаны быть:
- однозначными
- поисковыми
- смысловыми
- конкретными
Этикет:
- функции/методы: verb_noun (действие_объект)
- классы: PascalCase (Сущности)
- константы: UPPER_SNAKE_CASE
## пример:
```python
validate_user_session()
fetch_active_orders()
generate_invoice_pdf()
```

## 11. Обязательно выносить явно:
- конфиг
- env
- cache
- async
- retry
- side effects
- state mutations

# 12. Типизация и структуры данных
- >3 связанных параметров → структура/DTO
- смысловые примитивы → отдельные типы (UserId, Email)

# 13. Запрет магических значений
Магические:
- числа
- строки
- пути
- URL
- ключи
- таймауты
обязаны выноситься:
- в constants
- config
- enums

# 14. Dependency Discipline
Новая библиотека добавляется только если:
- решает сложную проблему
- экономит значительное время
- не дублирует существующее решение
Запрещено:
- подключать зависимость ради 1 функции

# 15. Fail Fast Rule
Функции обязаны:
- валидировать входные данные в начале
- быстро выбрасывать понятные ошибки
- не скрывать исключения
Поведение функции обязано быть очевидным.
Рекомендуемый docblock:
```python
"""
PURPOSE:
INPUT:
OUTPUT:
SIDE EFFECTS:
DEPENDENCIES:
ERRORS:
REUSABLE:
"""
```
# 16. Rule of Pure Functions
Предпочтение отдается pure-функциям.
Pure function:
- не меняет внешний state
- не зависит от скрытого state
- дает одинаковый результат при одинаковом input

# 17. One Level of Abstraction
Внутри одной функции запрещено смешивать:
- бизнес-логику
- SQL
- HTTP
- low-level implementation
Каждый уровень должен быть скрыт за понятными вызовами.

# 18. Anti-Hardcode Rule
Запрещено:
- хардкодить время
- пути
- URL
- токены
- настройки
Все внешние зависимости:
- передаются параметрами
- или берутся из config/env

# 19. Temporary Code
Только с пояснением:
# TEMP:
# reason:
# remove-after:

# 20. AI Navigation Layer
Крупные модули обязаны иметь:
```python
# module-purpose:
# entry-points:
# main-flows:
# related-modules:
```
# 21. Архитектурный манифест
Перед генерацией крупного функционала ИИ обязан:
1. Описать структуру модулей
2. Описать ответственности файлов
3. Описать связи компонентов
4. Только потом писать код

# 22. Rule of Predictability
Одна задача должна решаться одинаковым паттерном по всему проекту.
ИИ запрещено:
- использовать разные архитектурные подходы для одинаковых задач

# 23. AI Memory Anchors
Критически важные части системы обязаны помечаться:
```python
# critical: true
# architecture: auth-flow-v2
# related: token_cache
```
